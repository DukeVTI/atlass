Atlas is a cloud-brained, distributed AI agent with local workers on PC and mobile. She manages your email, calendar, finances, files, and communications autonomously � commanded through Telegram, alive 24/7 on a VPS, and extended to your devices through lightweight workers that give her eyes, hands, and a voice on every platform you use.






1. Executive Summary
Atlas is not a chatbot. She is an autonomous agent that lives in the cloud and reaches into your devices.
Atlas is a distributed, voice-and-text-first AI agent designed to serve as a private, autonomous executive assistant for a founder-operator. Unlike monolithic desktop assistants that hog resources and die when the laptop closes, Atlas is architecturally split: her brain lives permanently on a cloud VPS, her hands reach into your PC and phone through lightweight workers, and her voice reaches you through the messaging apps you already use � primarily Telegram, with WhatsApp as an outbound delivery channel.

The core design insight is that your laptop should not be the bottleneck. Atlas is always on, always reasoning, always watchful � regardless of whether your devices are powered on. When you open Telegram and say 'Atlas, what happened while I was asleep?', she already knows. She has been watching your inbox, monitoring your Paystack account, and tracking your calendar all night.

DESIGN PRINCIPLE
Speak once, get it done. Atlas is not a search engine with a voice interface. She is an agent that decomposes compound commands, executes them across multiple integrations, recovers from partial failures, and reports back with results � without hand-holding.

This document is the single source of truth for Atlas's architecture, feature set, integration plan, security model, and delivery roadmap. It covers both v1.0 (the initial production system) and v2.0 (the full autonomous agent). Every feature is tagged with its target version.

4. System Architecture
Three-tier distributed design: cloud brain, local workers, messaging interface.
Atlas is not a single application. It is a distributed system composed of three tiers that communicate over a secure, persistent WebSocket and task queue connection. Each tier has clearly defined responsibilities and can fail independently without taking down the entire system.

4.2 Tier 1 � VPS Brain
The VPS is Atlas's permanent residence. It runs 24/7, handles all intelligence and orchestration, and owns all third-party integrations that do not require local device access. This is the only tier that runs the LLM and manages memory.

Component	Responsibility	Technology
Telegram Gateway	Receive commands, send responses, manage bot state	python-telegram-bot / Telethon
WhatsApp Adapter	Outbound message delivery to contacts and clients	Baileys (WA Web) via Node.js microservice
Orchestration Engine	Intent parsing, task decomposition, tool routing, plan execution	Python + LangGraph or custom DAG
LLM Core	Reasoning, generation, tool selection, response synthesis	Anthropic Claude Haiku 4.5 (sole provider, no fallbacks)
Memory System	Short-term, episodic, factual, procedural memory layers	ChromaDB (vector) + PostgreSQL (structured)
Email Adapter	Gmail / IMAP read, compose, send, search, categorise	Google API Client + imaplib
Calendar Adapter	Google Calendar / CalDAV full CRUD + conflict detection	Google API Client + caldav
Paystack Adapter	Transactions, balance, transfers, customer data, webhooks	Paystack REST API + httpx
Web Search Adapter	Live web search, article fetch, content summarisation	Brave Search API / SerpAPI + Trafilatura
Worker Broker	Dispatch jobs to PC/mobile workers, collect results, manage timeouts	Redis + Celery
Scheduler	Cron-style recurring tasks (morning briefing, follow-up reminders)	Celery Beat
Security Module	Credential vault, permission gates, audit log	Keyring + cryptography + append-only SQLite log
API Server	Worker registration, health checks, webhook ingestion	FastAPI

4.3 Tier 2 � PC Worker
The PC worker is a lightweight background daemon running on the user's laptop. It has no LLM, no memory, and no intelligence. It is a dumb executor that receives structured job packets from the VPS, runs them, and returns results. It should consume negligible resources when idle (< 50MB RAM, < 0.5% CPU).

Capability	Details	Technology

File write / create	Write content to disk; new files or overwrite with confirmation	pathlib + confirmation gate
File search	Full-text keyword search; semantic search via local embeddings	ripgrep + sentence-transformers
File move / copy / delete	Filesystem operations with destructive-action gating	shutil + confirmation gate
Directory listing	List contents of a directory with metadata	os + pathlib
Script execution	Run shell or Python scripts; stream stdout/stderr to VPS	subprocess + asyncio
Screenshot	Capture current screen state; send to VPS for vision analysis	Pillow / mss
Application control	Open, close, or interact with desktop apps	subprocess + pyautogui
Document generation	Create PDF, DOCX, or Markdown from structured content	ReportLab + python-docx
Clipboard access	Read or write clipboard content	pyperclip

Browser Automation	Headless/headed browsing of authenticated sites	Playwright / Selenium
Local Dev & DB Access	Query local DBs, read Git repos, execute local test suites	psycopg2 / gitpython / subprocess
Hardware & IoT Control	Access local webcams, microphones, or network devices	OpenCV / pyaudio / requests

WORKER OFFLINE BEHAVIOUR
When the PC worker is offline (laptop closed, powered off, or worker process stopped), the VPS brain handles this gracefully. It queues any PC-bound tasks, reports their pending status to the user in Telegram, and executes them automatically when the worker reconnects. The user is notified of reconnection and queued task completion.

4.4 Tier 3 � Mobile Worker
The mobile worker is a lightweight app on the user's phone. Like the PC worker, it is a dumb executor. It gives Atlas access to phone-native capabilities that cannot be replicated from the cloud. It communicates with the VPS over a persistent WebSocket, authenticated by a device-specific token.

4.6 Data Flow � Standard Command
1.User sends text or voice note to Atlas Telegram bot.
2.If voice note: VPS transcribes audio using faster-whisper.
3.Orchestration engine parses intent, classifies task type, checks required tools.
4.Memory retrieval: fetch relevant episodic context and user preferences.
5.LLM generates action plan � ordered list of tool calls with inputs.
6.Executor runs action plan: VPS-local tools execute immediately; PC/mobile tasks dispatched via Redis to the appropriate worker.
7.Workers execute jobs, stream results back to VPS.
8.VPS assembles final response, synthesises narrative, sends to Telegram.
9.Full interaction (command, plan, tool calls, result) written to audit log and memory.


5. Command Interface � Telegram & WhatsApp
5.1 Telegram Bot � Primary Interface
Telegram is the primary interface through which all commands are sent to Atlas. The bot is a private bot � only the authorised user ID can interact with it. Any message from an unauthorised ID is silently ignored and logged.

5.1.1 Supported Input Types
?Text messages � natural language commands in any format or length.
?Voice notes � transcribed server-side; treated identically to text after transcription.
?Photos and images � passed to vision model for analysis; user can ask questions about them.
?Documents and PDFs � uploaded to VPS, processed, and available for reference in conversation.
?Forwarded messages � Atlas can act on forwarded emails, news articles, or messages.
?Inline bot commands � /start, /help, /status, /briefing, /stop (for aborting active tasks).

5.1.2 Confirmation UX
?Low-risk actions: Atlas executes immediately, then reports what she did.
?Medium-risk actions (send email, delete file): Atlas shows a summary with Telegram inline keyboard � [Confirm ?] [Cancel ?] [Edit ?].
?High-risk actions (Paystack transfer, overwrite important file): Atlas shows a detailed review card + requires the user to type a confirmation PIN or phrase.
?Timeout: if a confirmation is not received within 5 minutes, the action is automatically cancelled and Atlas reports this.

5.1.3 Atlas Response Modes
Mode	When Used	Format
Immediate	Task < 3s, result is brief	Single Telegram message
Streaming progress	Task takes > 3s	Atlas sends 'Working on it...' then edits message with result
Multi-part	Result is long (e.g. email summary)	Multiple messages with headers; offer to send as file
File response	Result is a document, PDF, spreadsheet	Telegram file upload with brief caption
Confirmation card	Medium/high risk action	Formatted message with inline keyboard
Error report	Task failed partially or fully	Plain explanation + what succeeded + what failed + options
Proactive alert	Atlas-initiated (payment, reminder)	Telegram message with context and suggested action

5.2 WhatsApp � Outbound Delivery
WhatsApp is not a command channel. Atlas uses it exclusively to send messages on the user's behalf to contacts and clients who communicate via WhatsApp. This keeps the risk of WhatsApp account bans minimal � Atlas is not running as a bot gateway but as an automated sender of user-initiated messages.

?Atlas can send text messages, voice notes, images, and documents to WhatsApp contacts.
?All WhatsApp sends are confirmed by the user in Telegram first.
?Contact matching: user says 'send Dana the project update on WhatsApp' � Atlas matches by name in contacts, confirms the number, then sends.
?Delivery status: Atlas reports read receipts and delivery confirmations back to Telegram.
?Virusia Academy use: Atlas sends scheduled lesson content to student WhatsApp groups on the user's behalf.

WHATSAPP RISK NOTE
WhatsApp's Terms of Service prohibit automated bulk messaging via unofficial APIs. Atlas's WhatsApp usage is intentionally low-volume and contact-targeted, not broadcast. The Baileys session runs on the user's own number and behaves like a power user, not a bot. The user accepts the small residual risk of account restriction.

6.1.2 Core Capabilities

6.1.3 Edge Cases

6.4 Local File System Operations (via PC Worker)  ? v1.0
6.4.1 Core Capabilities
?Read � open and return content of text, markdown, PDF, code, CSV, and JSON files.
?Write � create new files or overwrite existing with confirmation gate.
?Append � add content to the end of a file without overwriting.
?Move / copy � relocate files with path confirmation.
?Delete � moves to local trash bin first; hard delete only after second confirmation.
?Search � full-text keyword search (ripgrep); semantic similarity search via local embeddings.
?Directory listing � list contents of any scoped directory with file sizes and modified dates.
?Script execution � run shell or Python scripts; stream output back to Telegram; explain errors in plain English.
?Document generation � create PDFs, DOCX, or Markdown from dictated content.
?File watching � monitor a directory for new files; trigger Atlas actions on new arrivals (e.g. auto-summarise new PDFs in Downloads).
?Screenshot � capture screen and send to Atlas for visual analysis or record-keeping.

6.4.2 Security Boundaries
?Scoped roots � user defines allowed directories in settings; Atlas refuses access outside these.
?Hard-blocked paths � /etc, /sys, /proc, /boot, C:\Windows are permanently blocked regardless of settings.
?Sensitive file detection � warns before reading or transmitting files matching: *.pem, *id_rsa*, *.env, *secret*, *.key.
?Trash bin recovery � deleted files held in ~/.atlas-trash/ for 24 hours before permanent deletion.
?Execution sandboxing � scripts run in a subprocess with resource limits (CPU time, memory ceiling).

6.4.3 Edge Cases
?Binary files � detected and declined for read; metadata (size, type, modified date) returned instead.
?Large files (> 50MB) � chunked for summarisation; user warned of processing time before starting.
?Permission denied � OS-level permission errors surfaced in plain language with suggested fix.
?Symlink loops � circular symlink chains detected and broken; user alerted.
?Network drives / unmounted volumes � detected and reported before attempting access.
?Worker offline � PC tasks queued on VPS; executed on worker reconnect; user notified of delay.

9. Security, Privacy & Trust Model
Atlas is a cloud-first system. Security is not an afterthought � it is baked into every layer of the design.
9.1 Core Privacy Principles
?No telemetry � Atlas collects no analytics, usage data, or behavioural data for any third party.
?Credential vault � all API keys and OAuth tokens stored in the VPS keyring or an encrypted vault (Vault by HashiCorp, or a local AES-256 encrypted SQLite file). Never in .env files committed to git. Never in logs.
?Audit log � immutable, append-only log of every tool call Atlas makes, including timestamp, action taken, result, and whether user confirmation was granted. Stored in PostgreSQL with write-only access from the main process.
?Data minimisation � Atlas only fetches data it needs for the current task. No bulk data collection 'just in case'.
?User-controlled data deletion � user can instruct Atlas to wipe any or all memory categories on demand.

9.2 Confirmation Gate Matrix
Every action Atlas can take is classified by risk level. The confirmation gate defines what proof of user intent is required before execution. This cannot be overridden by a chat command � it is enforced in the executor layer.

Read local files (scoped)	None	No gate � execute and report
Draft email (not sent)	Low	No gate � Atlas shows draft for review
Create calendar event	Low	Atlas reads back details; user replies 'confirm'
Send email	Medium	Telegram inline keyboard: [Send ?] [Edit] [Cancel]
Reply to email	Medium	Telegram inline keyboard: [Send ?] [Edit] [Cancel]
Delete calendar event	Medium	Telegram inline keyboard + verbal 'yes delete it'
Post to social media	Medium	Telegram inline keyboard: [Post ?] [Edit] [Cancel]
Write/overwrite local file	Medium	Telegram inline keyboard with file path shown
Delete local file	Medium	Inline keyboard + file moved to trash first
Execute local script	Medium	Atlas shows script content; user confirms via inline keyboard
Send WhatsApp message to contact	Medium	Telegram inline keyboard with recipient + message preview

9.3 Telegram Security
?Allowlist-only � Atlas responds only to messages from the configured Telegram user ID. All other messages are silently dropped and logged.
?Bot token storage � Telegram bot token stored in VPS keyring only, never in source code or environment files committed to version control.
?Rate limiting � Atlas enforces a maximum of 60 commands per hour; alerts user if approaching limit due to scripted or accidental spam.
?Session hijack protection � Atlas detects abnormal patterns (commands from unusual times, unusual command sequences) and can temporarily lock itself pending user verification.

10. Agentic Engine & Memory System
10.1 Task Planning Model
When Atlas receives a compound command, she does not execute blindly. The orchestration engine decomposes the command into an ordered action graph � a directed acyclic graph (DAG) of tool calls where each node has defined inputs, expected outputs, and a success condition. This plan is executed node by node, with results from earlier nodes fed as inputs to later ones.

?Plan preview � for multi-step tasks involving 3+ actions or any medium/high-risk action, Atlas shows the plan to the user before executing: 'Here's what I'm going to do: (1) � (2) � (3) � Confirm?'
?Partial failure handling � if a node fails, Atlas marks it, continues with independent downstream nodes, and reports the partial result at the end.
?Circuit breaker � if the same tool call fails 3 consecutive times, Atlas aborts that branch and notifies the user.
?State persistence � multi-step task state is saved to PostgreSQL after each successful node, so Atlas can resume after a crash or disconnect.

10.2 Memory Architecture
Layer	Storage	Scope	TTL	Example Content
Working memory	Python dict (in-process)	Current request only	Single request	Parsed intent, tool call chain in progress
Conversation memory	Redis	Current session	Session end + 2h	Last 20 exchanges in the conversation
Episodic memory	ChromaDB (vector)	Long-term, semantic	90 days rolling	'Invoiced Dana ?150k on April 10', 'Samuel prefers WhatsApp'
Factual memory	PostgreSQL	Long-term, structured	Permanent	Contact list, project names, API key metadata, user prefs
Procedural memory	YAML files	Permanent	Until edited	Skill definitions, workflow templates, briefing format
Skill cache	Redis	Hot path	1 hour	Recently used skill definitions loaded in memory

10.3 Skill System
Skills are reusable, parameterised behaviour patterns defined in YAML. They allow Atlas to execute complex repeated tasks without re-reasoning from scratch every time. Skills are the primary mechanism for the user to customise Atlas's behaviour without writing code.

?Built-in skills (v1.0) � morning briefing, inbox triage, payment alert, meeting reminder, weekly revenue summary, daily end-of-day report.
?User-defined skills � 'Atlas, whenever I say project update, send a status email to the client with the following template�'
?Skill learning � Atlas can watch a user complete a multi-step task and offer to save it as a skill: 'You do this often. Want me to save it as a skill called Project Invoice?'
?Skill invocation � triggered by keyword match, schedule, webhook event, or explicit call.
?Skill parameters � skills accept runtime parameters injected at invocation ('invoice for ?50,000 to Dana').

Very long compound command	Atlas acknowledges receipt, shows decomposed plan, confirms before starting execution

15.2 Agentic & Multi-Step Tasks
Scenario	Atlas Behaviour
Mid-task API failure	Saves task state, reports exact failure point, offers retry / partial result / abort
LLM context window overflow	Orchestrator summarises earlier context and continues; user notified if significant compression occurs
Dependency failure (e.g. Calendar API down)	Continues independent steps; reports skipped step; offers to retry later
User disappears mid-confirmation	Task suspended to queue after 5min timeout; resumed on next user interaction
Infinite loop detection	Circuit breaker after 3 identical consecutive tool calls; Atlas alerts user and aborts
Tool returns unexpected data shape	Validator catches schema mismatch; Atlas reports data quality issue rather than propagating bad data
Task plan generates > 10 steps	Atlas presents summary and warns user of complexity; asks for confirmation before starting

15.3 Integration-Specific
Scenario	Atlas Behaviour

Paystack rate limit hit	Exponential backoff; batch remaining queries; notifies user of delay
Gmail sending limit reached	Queues remaining sends; notifies user; resumes when limit resets

16. Glossary
Term	Definition
Action graph / DAG	A directed acyclic graph of tool calls that represents Atlas's execution plan for a multi-step task
Agentic	Capable of autonomous multi-step task execution with tool use, planning, and error recovery
Baileys	An unofficial Node.js library that implements the WhatsApp Web protocol for programmatic messaging
Celery	A Python distributed task queue used for async job processing and scheduled tasks
ChromaDB	An open-source vector database used for semantic (embedding-based) memory storage and retrieval
Circuit breaker	A safety mechanism that aborts a failing operation after N consecutive failures to prevent infinite loops
Cloudflare Tunnel	A service that exposes local servers to the internet without port forwarding, used for webhook ingestion
Confirmation gate	A mandatory user approval step enforced at the executor layer before a high-risk action runs
Episodic memory	Long-term storage of past events and interactions, searchable by semantic meaning
LangGraph	A framework for building stateful, multi-step agent workflows as directed graphs
PC Worker	A lightweight background daemon on the user's laptop that executes filesystem and local app tasks on behalf of the VPS
Procedural memory	Stored skill definitions and workflow templates that tell Atlas how to perform recurring tasks
Scoped roots	User-defined directories that the PC worker is permitted to access; all others are blocked
Skill	A reusable, parameterised action pattern defined in YAML, triggered by keyword, schedule, or event
STT	Speech-to-Text � converts audio input (voice notes) into a text transcript for processing
TTS	Text-to-Speech � converts Atlas's text responses into spoken audio output
VPS Brain	The cloud VPS server that hosts Atlas's orchestration engine, LLM, memory, and integrations � always on
WebSocket	A persistent, bidirectional network connection used for real-time communication between the VPS and local workers
Working memory	In-process, single-request context that is discarded after each task completes
